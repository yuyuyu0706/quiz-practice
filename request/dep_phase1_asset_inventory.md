# DEP Phase 1 資材棚卸しメモ

作成日: 2026-03-22  
対象リポジトリ: `quiz-practice`

## 1. 前提と整理方針

- 今回は **実装変更ではなく棚卸しと分類** を目的とする。
- 判定軸は以下の 4 つとした。
  - **依存性**: DEA依存 / DEP依存 / 共通
  - **方針**: 共通化候補 / 分離維持 / 保留
  - **リファクタ必要性**: 高 / 中 / 低
  - **対応タイミング**: 今 / 後 / 不要
- 「今」は、次フェーズの安全な着手前に判断・設計だけでも先に固めたいものを指す。
- 「後」は、現時点では分離のまま運用して支障が少なく、後続フェーズで扱えばよいものを指す。
- 「不要」は、現状のままで目的に対して十分で、直近の整理対象に含めなくてよいものを指す。

## 2. 現状サマリ

- `dea-quiz-app` と `dep-quiz-app` は、現状では **HTML/CSS/JavaScript のほぼ同型コピー** で運用されている。
- 実差分として確認できたのは、主に **タイトル/見出し文言** と **localStorage キー名**、および `app.js` 内の `app` 識別子である。
- テスト基盤は **DEA 側中心** に整備されており、Playwright 設定やヘルパー構造は将来的に DEP へ横展開しやすい。
- 一方で、**questions.json バリデーション** と **GitHub Actions ワークフロー** はまだ DEP を十分カバーしていない。特に `.github/workflows/` 配下は現時点で未作成だった。

## 4. 重点観点ごとの整理

### 4.1 共通化しやすい層

#### A. クイズ実行ロジック（最優先）
共通化しやすいのは `app.js` に集約されている以下のロジック群である。

- 採点ロジック
- 出題順生成（通常 / ランダム / wrongOnly / bookmarks）
- 中断 / 再開
- explanation / references 表示
- 結果画面表示
- モバイル向け副操作 UI
- キーボード操作

理由:

- DEA/DEP の `app.js` は、確認できた差分が **localStorage キー** と **アプリ識別子** ほぼのみである。
- UI 仕様も現時点ではほぼ一致している。
- 重複実装のまま今後機能追加すると、DEA と DEP の修正漏れリスクが高い。

推奨方針:

- いきなり shared ディレクトリを作るのではなく、まずは「差分一覧」を固定化する。
- 次フェーズで、`appConfig` を注入して動く単一ロジック化を検討する。
  - 例: `storageKeys`, `appTitle`, `questionPath`, `labels`

#### B. questions.json バリデーションルール
共通化しやすいのは **バリデーション規則** であり、**問題データ本体ではない**。

- 必須フィールド
- `choices` が `A/B/C/D`
- `answer` が `A/B/C/D`
- `id` 一意
- `references` 構造

推奨方針:

- `scripts/validate-dea-questions.mjs` を直接 DEP に複製するより、
  - 共通バリデーション関数
  - 対象ファイルだけ切り替える実行入口
  という形に寄せると保守性が高い。

#### C. Playwright 基盤
以下は既に共通基盤候補として十分機能している。

- `playwright.config.ts`
- テスト起動方法
- 失敗時 artifact 方針
- desktop/mobile プロジェクト定義

推奨方針:

- 共通基盤は維持しつつ、DEP 用 spec を段階的に追加する。
- spec ファイルは試験別に分けたまま、helper だけ共通化する構成がバランスがよい。

### 4.2 まだ分けて持つべき層

#### A. 問題データ (`questions.json`)
これは DEA / DEP で完全に分離維持が妥当。

- 試験範囲が異なる
- セクション構成やタグの意味が変わりうる
- 将来的な更新頻度・レビュー観点も異なる

#### B. localStorage キー
ロジックが共通化されても、保存キーは分離を維持すべき。

- 同一ブラウザで履歴が混ざる事故を防げる
- Associate と Professional を並行学習するケースに適している

#### C. README / 学習導線 / 試験名
README やアプリのタイトル・説明文は、利用者導線の観点で分離維持が自然。

- 試験名
- 起動例
- 学習方針
- データ投入状況

### 4.3 今は保留でよい層

以下は共通化余地があるが、現時点では保留が適切。

- CSS の完全共通化
- HTML のテンプレート化 / コンポーネント化
- `shared/` ディレクトリ新設
- 画面描画ロジックの全面抽象化
- Professional 向け拡張を前提にした先回りリファクタ

理由:

- まだ DEP 側の機能差分が十分に見えていない。
- 先に UI 層まで強く抽象化すると、後から差分が出たときに戻しコストがかかる。
- まずは **ロジック層・検証層の重複** から着手した方が安全。

## 5. リファクタリング必要性の整理

### 高

- `dea-quiz-app/app.js`
- `dep-quiz-app/app.js`

理由:

- ほぼ同一コードが二重管理になっている。
- 片側だけ修正される事故が今後起こりやすい。
- クイズ機能追加時の変更箇所が増え続ける。

### 中

- `scripts/validate-dea-questions.mjs`
- `package.json`
- `tests/e2e/helpers.ts`
- `tests/README.md`
- 両アプリの `index.html` / `styles.css`

理由:

- 直ちに破綻はしないが、DEP を本格運用すると二重管理が目立つ。
- ただし UI 層は抽象化の設計余地を見てからでも遅くない。

### 低

- 各 `questions.json`
- 各 `README.md`
- DEA 向け各 spec

理由:

- 試験固有資材として分離維持の合理性が高い。
- リファクタより、今後の内容拡充や運用更新の方が本質的。

## 6. 全体所見

### 6.1 共通化しやすいのは「ロジック層」と「検証基盤」
最も共通化効果が高いのは、見た目ではなく以下である。

- `app.js` のクイズ進行ロジック
- `questions.json` のバリデーションルール
- Playwright の共通 helper / 設定

この 3 層は、DEA と DEP の仕様が今のところほぼ一致しているため、差分を設定値に押し込めやすい。

### 6.2 まだ分けるべきなのは「試験固有資材」
分けたままの方が安全なのは以下である。

- `questions.json`
- localStorage キー
- README
- アプリタイトルや試験名
- 将来的なカテゴリ / ドメイン定義

つまり、**共通化すべきなのは土台、分けるべきなのは中身** という整理が最も自然である。

### 6.3 UI 層の共通化は急がなくてよい
HTML/CSS は現状かなり似ているが、ここを最初に shared 化すると、今後 DEP 独自要件が出たときに柔軟性を失う可能性がある。
そのため、次段階では UI 共通化よりも **JS ロジック / テスト / バリデーション** の整理を優先した方が安全である。

### 6.4 CI は今後の重要な補強点
テスト README には CI 連携方針が書かれているが、現時点で `.github/workflows/` 配下の実ファイルは存在しなかった。
そのため、継続開発に備えるなら、次フェーズのどこかで以下を明文化・実装したい。

- DEA の自動検証固定化
- DEP 追加時の検証入口追加
- artifact 保存方針の定着

## 7. 次フェーズへの提案

### 推奨着手順

1. **questions バリデーションの共通化設計** を先に固める  
   - 実データは分けたまま、検証ルールだけ共通化する。
   - DEP に本番問題を入れ始める前の安全網になる。

2. **`app.js` の差分棚卸しをもう一段だけ細かく実施** する  
   - 具体的には「設定化できる差分」と「分岐が必要な差分」を分離する。
   - そのうえで単一ロジック化に進むかを判断する。

3. **Playwright helper の共通化準備** を行う  
   - spec は DEA/DEP 別のまま維持。
   - helper だけ appPath・タイトル可変に寄せる。

4. **その後に DEP の問題投入 / UI 微調整へ進む**  
   - 先に最低限の安全網を作っておくと、Professional 版拡張で壊しにくい。

### 「共通化を先にやるべきか」への回答

**全面共通化を先にやる必要はない。** ただし、以下の限定的な共通化準備は先にやる価値が高い。

- バリデーションルール
- テスト helper
- `app.js` の設定差分整理

逆に、以下は後回しでよい。

- CSS 共通化
- HTML テンプレート化
- shared ディレクトリへの大移動

### 「Professional版の問題投入やUI調整を先にやるべきか」への回答

**DEP の問題投入自体は進めてよいが、その前に最低限の検証導線を整えるのが安全** である。
特におすすめなのは、以下の順序。

- 先にデータ検証入口を DEP でも使える形にする
- 次に DEP の問題投入を進める
- 実際に差分が出てきた段階で、JS 共通化の範囲を最終判断する

## 8. 判断まとめ（短縮版）

- **最優先の共通化候補**: `app.js`、questions バリデーション、Playwright 共通 helper / config
- **分離維持が妥当**: `questions.json`、localStorage キー、README、試験名・導線
- **今は保留でよい**: CSS 完全共通化、HTML コンポーネント化、shared ディレクトリ化
- **次に安全な一手**: DEP 問題投入前に、バリデーションとテスト基盤の再利用方針を整理する
