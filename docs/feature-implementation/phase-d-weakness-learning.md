# 弱点分析 機能実装書

## 本文書について

本文書は、DEPクイズアプリの弱点分析について、機能群が必要とされた背景、各機能の関係、設計思想、現在の到達点、今後の発展余地を理解するための機能実装書である。

弱点分析では、学習履歴から弱点を見つけるだけでなく、分析結果を確認し、弱点に沿って復習し、必要に応じて学習履歴を安全に切り替えることで、次の学習判断につなげる循環を整備した。
本文書は、その循環を初見の読者にも追えるように説明することを目的とする。

本文書では、弱点分析を初見読者が理解しやすい順序で説明するため、文書全体のストーリーラインと章構成を定義する。
各章の詳細本文、参照先、「書く / 書かない」、完成条件は後続の文書詳細化で確定する。

### 目的

本文書の目的は、弱点分析が、なぜ必要とされ、どのような機能群として実装され、各機能がどのような学習循環を形成し、どの設計思想によって支えられているかを理解できるようにすることである。

読者が本文書を読んだ後に、少なくとも次の流れを説明できる状態を目指す。

1. 弱点分析が解決しようとした課題。
2. 弱点分析だけでは学習行動につながり切らなかった理由。
3. 対象問題一覧、弱点起点の復習、学習履歴リセットが必要になった理由。
4. 各機能がどのように連携して、分析から復習、再分析へ向かう学習フィードバックループを作るか。
5. 現行実装で重視した設計判断と、その背景にある考え方。
6. 将来、時系列分析・クロス分析・学習効果測定・AI推薦などへ発展させる際に、どの観点を引き継ぐべきか。

本文書は、Issue、PR、コード、Unit Test、E2E、E2Eカタログの内容を別形式で網羅的に再掲するものではない。
詳細な要件、変更差分、正確な実装契約、保証される振る舞いは、それぞれの成果物を正本として参照する。

### 想定読者

#### 主読者

- 将来、弱点分析や弱点学習機能を改修する開発者。
- 特に、コードや個別Issueを読む前に、機能全体の意図、構造、設計判断の背景を把握したい開発者。
- この読者に対しては、関数や画面を単独で説明するのではなく、学習履歴を次の学習判断に利用するために、なぜその機能が必要になったのかを中心に説明する。

#### 副読者

- DEPクイズアプリの機能体系や弱点分析の成果を把握したい関係者。
- 詳細Issueをすべて追わなくても、弱点分析がどのような価値を追加し、既存のクイズ学習体験をどのように拡張したのかを理解できることを重視する。

#### 非対象の読者像

本文書は、次の用途を主目的にしない。

- 操作手順だけを知りたい利用者向けマニュアル。
- API、関数、型、storage keyの完全な仕様だけを調べるコードリファレンス。
- 全PR差分や全Issue本文を追跡する変更履歴。
- Unit TestやE2Eケースを逐語的に列挙するテスト一覧。

### 文書の位置付け

弱点分析に関する成果物は複数に分かれているため、本文書では次の役割分担を前提にする。

| 成果物          | 主な役割                                                    |
| --------------- | ----------------------------------------------------------- |
| Issue           | 要件、背景、判断経緯、受入条件を記録する。                  |
| PR              | 実際に変更した内容とレビュー単位を記録する。                |
| コード          | 現行実装の正本として、実際の動作と契約を示す。              |
| Unit Test / E2E | 保証される振る舞いと回帰防止対象を示す。                    |
| E2Eカタログ     | E2Eテスト責務の正本として、何をどの粒度で確認するかを示す。 |
| 機能実装書      | 機能の意味、目的、関係、設計思想、現在地、展望を説明する。  |

機能実装書では、「どう実装したか」よりも「なぜそう実装したか」を優先する。
ただし、設計思想を理解するために必要な範囲では、`depQuizProgress`、`depQuizSettings`、`depQuizActiveSession`、`buildWeaknessAnalysis()`、`weaknessReview`のような実装名を扱うことがある。

一方で、関数の全引数・全返却値、全プロパティ一覧、全Unit Test / E2Eケース、PRごとの差分、Issue本文の全文再掲は本文の中心にしない。
これらは、必要に応じて各正本へ読者を誘導する。

### 記述方針

#### 目的から書く

各機能は、実装名や関数名から説明を始めない。
まず、その機能が何を解決するために必要だったのかを説明する。

例えば、弱点分析は「`buildWeaknessAnalysis()`を追加した」ことではなく、「蓄積された学習履歴を次の学習判断に利用できるよう、全体・Section・誤答理由の観点から弱点を分析する機能を設けた」こととして説明する。

#### 学習循環の中で説明する

機能単体の追加事実ではなく、学習体験上の役割を中心に説明する。
対象問題一覧であれば、単に一覧画面を追加したと説明するのではなく、分析結果を確認し、復習対象に納得し、弱点起点の復習へ進むための接続点として説明する。

本文書では、分析、対象確認、復習、再分析、必要に応じた履歴リセットがどのように結び付き、学習フィードバックループを形成するかを重視する。

#### 実装詳細は必要最小限にする

設計判断を理解するために必要な技術要素は扱うが、コード解説書にはしない。
実装名、storage key、mode名などは、文脈を理解する補助として利用し、詳細な仕様確認はコードやテストに委ねる。

#### 現行実装と将来展望を分離する

現行実装、実装済みの事実、将来案を同じ表現レベルで混在させない。

現行実装として扱う内容には、全体・Section・誤答理由タグ別の分析、対象問題一覧、`weaknessReview`、学習履歴リセットなどが含まれる。
将来展望として扱う内容には、時系列分析、クロス分析、学習効果測定、AI推薦などが含まれる。

将来展望を書く場合は、現時点で実装済みであるかのように読める表現を避け、現行機能から自然に発展し得る方向性として明確に分離する。

#### Issue番号を本文の主語にしない

本文では、「どのIssueで実装されたか」ではなく、「どの機能が何を解決するか」を主語にする。
IssueやPRは、詳細確認のための参照先として扱う。

例えば、「#197で実装した」ではなく「弱点分析」、「#253で実装した」ではなく「弱点起点の復習」のように、機能や学習体験上の役割を中心に記述する。

#### 初見読者がIssue階層を知らなくても読めるようにする

本文は、Issue階層や過去の議論を知らない読者でも、弱点分析の主要な意味、機能関係、設計思想を理解できる自己完結性を持たせる。

必要な背景は本文中で要約し、詳細な判断経緯や受入条件を確認したい場合にIssueやPRを参照できる構成にする。

### 目次設計で使った判断基準

目次案比較では、次の観点を判断基準にした。

- 初見読者が理解しやすいか。
- Issue階層を知らなくても読めるか。
- 実装目的が中心になっているか。
- 学習フィードバックループが理解できるか。
- 詳細仕様の再掲になっていないか。
- 章間重複を避けられるか。
- 現行実装と将来展望を分離できるか。
- 今後の分析・可視化・学習支援の展望まで自然につながるか。
- 後続の章別詳細化へ分割しやすいか。

この判断基準に基づき、学習体験とプロダクト概念を中心にした章構成を採用する。

## 第1章 弱点分析の位置付け

本章では、弱点分析が必要になった背景と、クイズを「解く・保存する」体験から、学習履歴を使って次の学習行動を決める体験へ拡張した意味を説明する。

DEPクイズアプリの基本的な学習体験は、問題を解き、その結果を保存し、学習履歴を蓄積することから始まる。この流れにより、利用者は自分がどの問題に取り組み、どのような結果を残したかを後から確認できるようになった。学習履歴は、単発の解答結果を失わずに積み上げるための土台であり、継続的な学習を支える重要な情報である。

しかし、履歴が蓄積されているだけでは、次に何を学ぶべきかを判断するには十分ではない。正解数や過去の解答結果を見られても、どの範囲でつまずきが多いのか、どの誤答傾向を優先して復習すべきか、今の学習で注目すべき対象は何かを利用者が自力で読み解く必要が残る。つまり、「解く」「保存する」だけでは、学習履歴が次の学習行動へ自然につながり切らないという課題があった。

弱点分析は、この課題を解決するために、蓄積された学習履歴を振り返りの材料から学習判断の材料へ変える役割を持つ。学習履歴を分析することで、利用者は単に過去の結果を見るのではなく、自分がどの観点を重点的に見直すべきかを把握しやすくなる。ここで重視したのは、分析結果そのものを目的にすることではなく、分析を通じて次の学習行動を決めやすくすることである。

この考え方により、学習体験は、問題を解いて結果を保存するだけの流れから、蓄積された履歴を分析し、弱点を見つけ、次の学習行動を決める流れへ拡張される。図1は、この変化を「従来の学習体験」と「弱点分析を導入した学習体験」の対比として示している。

![学習履歴を次の学習判断に変える弱点分析の位置付け](images/phase-d-weakness-learning-section1.png)

弱点分析によって、学習履歴は「過去の結果の記録」から「次に取り組むべき内容を考えるための入口」へ変わる。図のように、利用者は、蓄積された結果を手がかりに弱点を見つけ、復習の優先度を考え、次の学習へ進むことができる。

一方で、弱点分析の結果を見るだけでは、学習行動は完結しない。分析で弱点が見えても、どの問題が対象なのかを確認できなければ復習に移りづらく、復習した結果が学習履歴へ戻らなければ改善状況を再び判断できない。また、学習を一区切りにして新しい周期へ移りたい場面では、履歴をどのように扱うかも学習体験の一部になる。

そのため、弱点分析は単独の閲覧機能ではなく、対象確認、弱点起点の復習、進捗更新、必要に応じた学習履歴リセットと組み合わせて意味を持つ。第2章では、これらの機能がどのように接続され、学習履歴を次の行動へ変えるフィードバックループを形成するかを説明する。

## 第2章 弱点分析を起点とする学習フィードバックループ

本章では、分析、対象確認、復習、進捗更新、再分析、必要に応じた学習履歴リセットが、一つの学習循環としてどう接続されるかを説明する。

弱点分析は、学習履歴から弱点を見つける入口である。ただし、分析結果を表示するだけでは、利用者が実際に復習し、進捗を更新し、改善後の状態を確認するところまでは到達しない。弱点分析を学習行動へつなげるには、分析結果の確認から復習、履歴更新、再分析までを一連の流れとして扱う必要がある。

この流れは、図2のように、学習履歴を起点に弱点分析、対象確認、弱点起点の復習、進捗更新、再分析へ戻る学習フィードバックループとして整理できる。図は、通常の復習サイクルと、必要に応じて学習履歴リセットから新しい学習周期へ移る分岐を一体で示している。

![弱点分析から対象確認、復習、進捗更新、再分析へつながる学習フィードバックループ](images/phase-d-weakness-learning-section2.png)

まず、既存の学習履歴が弱点分析の入力になる。分析は、蓄積された結果をもとに、利用者が次に注目すべき弱点を判断するための情報を提示する。ここでの分析結果は、復習へ進むための出発点であり、最終的な到達点ではない。

次に、対象確認が分析と復習の間をつなぐ。分析結果だけでは、利用者が具体的にどの問題を復習するのかを十分に把握できない場合がある。対象問題一覧を通じて、弱点として扱われた問題群を確認できるようにすることで、利用者は復習対象に納得したうえで次の行動へ進める。対象確認は、分析結果を抽象的な傾向から具体的な復習対象へ橋渡しする役割を持つ。

その後、弱点起点の復習によって、利用者は確認した対象に沿って再度問題に取り組む。復習の結果は、独立した一時的な記録として閉じるのではなく、既存の学習履歴へ戻る。これにより、復習で得られた正誤や進捗が次回以降の判断材料に含まれ、最新の履歴をもとに再び弱点分析できる。

この循環により、弱点分析は一度だけ見るレポートではなく、学習履歴を更新しながら繰り返し利用する判断の起点になる。利用者は、分析で弱点を見つけ、対象を確認し、復習し、その結果を履歴へ戻し、更新された履歴から再分析することで、学習状態の変化を次の行動へ反映できる。

学習履歴リセットは、この循環を毎回通過する通常工程ではない。リセットは、学習を一区切りにしたい場合や、蓄積された履歴を前提にした判断から新しい学習周期へ切り替えたい場合に使う分岐である。したがって、弱点分析から常にリセットへ進むのではなく、通常は対象確認と復習を通じて履歴を更新し、必要な場面でだけ新しい周期への切替手段として利用する。

このように、弱点分析、対象問題一覧、弱点起点の復習、学習履歴リセットは、ばらばらに追加された機能ではない。それぞれが、学習履歴を起点にして、弱点を見つけ、復習し、更新された状態を再び判断するためのフィードバックループを構成している。

次章では、このループの起点である弱点分析が、蓄積された学習履歴をどのように学習判断へ変換するのかを説明する。

## 第3章 学習履歴を学習判断へ変える ― 弱点分析の仕組み

本章では、蓄積された学習履歴を、次に何を学ぶべきか判断できる情報へ変換する弱点分析の目的と役割を説明する。

学習履歴は、利用者がどの問題に取り組み、どの結果を残したかを示す重要な記録である。しかし、履歴が増えるほど、単に過去の正誤を眺めるだけでは、次にどこを見るべきか、何を優先して復習すべきかを判断しづらくなる。弱点分析は、この履歴の読み解き負荷を下げ、過去の回答記録を次の学習判断に使える形へ変換するための入口である。

図3は、学習履歴を弱点分析へ渡し、全体像、Sectionごとの偏り、誤答理由の傾向、次に注目すべき対象へ整理していく考え方を示している。本文では各観点の細かな列挙を繰り返さず、図が示す分析の流れを前提に、弱点分析を学習判断へつなげるための設計意図に絞って説明する。

![学習履歴を学習判断へ変換する弱点分析の仕組み](images/phase-d-weakness-learning-section3.png)

弱点分析で重要なのは、低い正答率を機械的にすべて同じ弱点として扱わないことである。未学習に近い対象、回答数が少なく判断材料が不足している対象、十分に取り組んだうえで誤答が残る対象は、学習上の意味が異なる。これらを区別して考えることで、利用者は「まだ情報が足りないから学習を進める」のか、「明確な弱点として復習する」のかを判断しやすくなる。

また、弱点分析は過去のある時点で固定された成績表ではない。復習や通常学習によって履歴が更新されれば、次に見るべき内容も変わり得る。分析は最新の学習履歴を前提に、現在の状態から次の判断を導くために利用される。

このように、弱点分析は、学習履歴を「過去に何をしたか」の記録にとどめず、「次に何を見るべきか」「何を優先すべきか」を考える材料へ変える。ただし、何を学ぶべきかが分かっても、具体的にどの問題が対象なのかを確認し、納得して復習行動へ移れなければ、学習は前に進まない。次章では、分析結果を具体的な対象確認と復習へつなぐ役割を説明する。

## 第4章 分析を学習行動へつなぐ ― 対象問題一覧と弱点起点の復習

本章では、分析結果を確認し、対象問題に納得したうえで復習を開始し、その結果を再び学習履歴へ戻す流れを説明する。

弱点分析によって、利用者は次に注目すべき領域や誤答傾向を把握しやすくなる。しかし、分析結果を見ただけでは、実際にどの問題を復習するのか、なぜその問題が対象になっているのかを十分に確認できない場合がある。分析を学習行動へ変えるには、抽象的な傾向を具体的な復習対象として確認できる接続点が必要になる。

図4は、分析結果を対象問題一覧で確認し、弱点起点の復習へ進み、その結果が既存の学習履歴へ戻る流れを示している。対象確認から復習、進捗反映までの手順は図に委ね、本文ではこの流れが学習体験上どのような意味を持つかを整理する。

![分析結果を対象問題一覧と弱点起点の復習へつなぐ流れ](images/phase-d-weakness-learning-section4.png)

対象問題一覧は、分析結果と学習行動の間にある橋渡しである。利用者は、選んだ弱点条件に対応する問題群を確認することで、分析が示した傾向を「実際に取り組む問題」として受け止められる。対象が明示されることで、弱点分析の結果は一方的に提示されるだけの情報ではなく、利用者が確認し、理解し、納得して復習へ進むための材料になる。

弱点起点の復習は、対象問題一覧で確認した条件をもとに、既存のクイズ体験へ接続する。復習結果は専用の一時履歴に閉じ込めず、既存の学習履歴へ戻す。これにより、弱点起点で解いた問題の正誤や進捗が次回以降の分析材料に含まれ、分析、対象確認、復習、履歴更新、再分析という循環が成立する。

対象問題一覧と弱点起点の復習は、分析を見て終わるための機能ではなく、分析を学習行動へ変えるための接続である。一方で、学習履歴は復習や通常学習によって更新され続けるため、試験前後や学習方針の変更など、これまでの履歴を一区切りにして新しい周期へ移りたい場面もある。次章では、そのような場面で学習履歴をどう扱うかを説明する。

## 第5章 学習サイクルを切り替える ― 学習履歴リセット

本章では、学習履歴を永久的な成績表として固定せず、保持すべき資産を残しながら新しい学習サイクルへ移行する考え方を説明する。

弱点分析は、蓄積された学習履歴をもとに現在の状態を判断する。学習を続けるほど履歴は増え、分析はその履歴を材料に、今の弱点や復習候補を示す。一方で、学習履歴をいつまでも同じ周期の評価情報として扱い続けることが、常に最適とは限らない。試験が終わった後に次の目標へ切り替える場合や、学習方針を大きく変える場合には、これまでの履歴を一区切りにして新しい周期へ移る方が分かりやすいことがある。

図5は、学習履歴リセットを、すべてを消す操作ではなく、評価履歴を一区切りにしながら学習資産を保持して次の周期へ移るための分岐として示している。リセット対象と保持対象の詳細な対応や周期の流れは図に委ね、本文ではリセットを学習体験上どのように位置付けるかを説明する。

![学習履歴を一区切りにし学習資産を残して新しい周期へ移るリセットの考え方](images/phase-d-weakness-learning-section5.png)

学習履歴リセットは、単にデータを全削除するための機能ではない。これまでの評価履歴を一区切りにし、次の学習を新しい前提で始められるようにするための学習周期の切替手段である。リセット後は、新しく蓄積される履歴をもとに、再び弱点分析と復習の循環を始められる。

ここで重要なのは、評価履歴と学習資産を同じものとして扱わないことである。学習結果や誤答理由は現在の状態を判断するための材料であり、新しい周期へ切り替えるときには一区切りにする対象になる。一方で、メモ、ブックマーク、設定のような学習資産は、利用者が学習を進める中で育ててきた補助情報や環境であり、新しい周期でも足場として残す意味がある。

このように、学習履歴リセットは、過去の成績に引きずられず新しい周期を開始しつつ、学習のために蓄積してきた資産は引き続き利用できるようにするための分岐である。ここまで見てきた弱点分析、対象問題一覧、弱点起点の復習、学習履歴リセットには、共通する設計思想がある。次章では、これらの機能を支える主要な設計思想を整理する。
