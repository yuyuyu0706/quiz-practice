# 弱点分析 機能実装書

## 本文書について

本文書は、DEPクイズアプリの弱点分析について、機能群が必要とされた背景、各機能の関係、設計思想、現在の到達点、今後の発展余地を理解するための機能実装書である。

弱点分析では、学習履歴から弱点を見つけるだけでなく、分析結果を確認し、弱点に沿って復習し、必要に応じて学習履歴を安全に切り替えることで、次の学習判断につなげる循環を整備した。
本文書は、その循環を初見の読者にも追えるように説明することを目的とする。

本文書では、弱点分析を初見読者が理解しやすい順序で説明するため、文書全体のストーリーラインと章構成を定義する。
各章の詳細本文、参照先、「書く / 書かない」、完成条件は後続の文書詳細化で確定する。

### 目的

本文書の目的は、弱点分析が、なぜ必要とされ、どのような機能群として実装され、各機能がどのような学習循環を形成し、どの設計思想によって支えられているかを理解できるようにすることである。

読者が本文書を読んだ後に、少なくとも次の流れを説明できる状態を目指す。

1. 弱点分析が解決しようとした課題。
2. 弱点分析だけでは学習行動につながり切らなかった理由。
3. 対象問題一覧、弱点起点の復習、学習履歴リセットが必要になった理由。
4. 各機能がどのように連携して、分析から復習、再分析へ向かう学習フィードバックループを作るか。
5. 現行実装で重視した設計判断と、その背景にある考え方。
6. 将来、時系列分析・クロス分析・学習効果測定・AI推薦などへ発展させる際に、どの観点を引き継ぐべきか。

本文書は、Issue、PR、コード、Unit Test、E2E、E2Eカタログの内容を別形式で網羅的に再掲するものではない。
詳細な要件、変更差分、正確な実装契約、保証される振る舞いは、それぞれの成果物を正本として参照する。

### 想定読者

#### 主読者

- 将来、弱点分析や弱点学習機能を改修する開発者。
- 特に、コードや個別Issueを読む前に、機能全体の意図、構造、設計判断の背景を把握したい開発者。
- この読者に対しては、関数や画面を単独で説明するのではなく、学習履歴を次の学習判断に利用するために、なぜその機能が必要になったのかを中心に説明する。

#### 副読者

- DEPクイズアプリの機能体系や弱点分析の成果を把握したい関係者。
- 詳細Issueをすべて追わなくても、弱点分析がどのような価値を追加し、既存のクイズ学習体験をどのように拡張したのかを理解できることを重視する。

#### 非対象の読者像

本文書は、次の用途を主目的にしない。

- 操作手順だけを知りたい利用者向けマニュアル。
- API、関数、型、storage keyの完全な仕様だけを調べるコードリファレンス。
- 全PR差分や全Issue本文を追跡する変更履歴。
- Unit TestやE2Eケースを逐語的に列挙するテスト一覧。

### 文書の位置付け

弱点分析に関する成果物は複数に分かれているため、本文書では次の役割分担を前提にする。

| 成果物          | 主な役割                                                    |
| --------------- | ----------------------------------------------------------- |
| Issue           | 要件、背景、判断経緯、受入条件を記録する。                  |
| PR              | 実際に変更した内容とレビュー単位を記録する。                |
| コード          | 現行実装の正本として、実際の動作と契約を示す。              |
| Unit Test / E2E | 保証される振る舞いと回帰防止対象を示す。                    |
| E2Eカタログ     | E2Eテスト責務の正本として、何をどの粒度で確認するかを示す。 |
| 機能実装書      | 機能の意味、目的、関係、設計思想、現在地、展望を説明する。  |

機能実装書では、「どう実装したか」よりも「なぜそう実装したか」を優先する。
ただし、設計思想を理解するために必要な範囲では、`depQuizProgress`、`depQuizSettings`、`depQuizActiveSession`、`buildWeaknessAnalysis()`、`weaknessReview`のような実装名を扱うことがある。

一方で、関数の全引数・全返却値、全プロパティ一覧、全Unit Test / E2Eケース、PRごとの差分、Issue本文の全文再掲は本文の中心にしない。
これらは、必要に応じて各正本へ読者を誘導する。

### 記述方針

#### 目的から書く

各機能は、実装名や関数名から説明を始めない。
まず、その機能が何を解決するために必要だったのかを説明する。

例えば、弱点分析は「`buildWeaknessAnalysis()`を追加した」ことではなく、「蓄積された学習履歴を次の学習判断に利用できるよう、全体・Section・誤答理由の観点から弱点を分析する機能を設けた」こととして説明する。

#### 学習循環の中で説明する

機能単体の追加事実ではなく、学習体験上の役割を中心に説明する。
対象問題一覧であれば、単に一覧画面を追加したと説明するのではなく、分析結果を確認し、復習対象に納得し、弱点起点の復習へ進むための接続点として説明する。

本文書では、分析、対象確認、復習、再分析、必要に応じた履歴リセットがどのように結び付き、学習フィードバックループを形成するかを重視する。

#### 実装詳細は必要最小限にする

設計判断を理解するために必要な技術要素は扱うが、コード解説書にはしない。
実装名、storage key、mode名などは、文脈を理解する補助として利用し、詳細な仕様確認はコードやテストに委ねる。

#### 現行実装と将来展望を分離する

現行実装、実装済みの事実、将来案を同じ表現レベルで混在させない。

現行実装として扱う内容には、全体・Section・誤答理由タグ別の分析、対象問題一覧、`weaknessReview`、学習履歴リセットなどが含まれる。
将来展望として扱う内容には、時系列分析、クロス分析、学習効果測定、AI推薦などが含まれる。

将来展望を書く場合は、現時点で実装済みであるかのように読める表現を避け、現行機能から自然に発展し得る方向性として明確に分離する。

#### Issue番号を本文の主語にしない

本文では、「どのIssueで実装されたか」ではなく、「どの機能が何を解決するか」を主語にする。
IssueやPRは、詳細確認のための参照先として扱う。

例えば、「#197で実装した」ではなく「弱点分析」、「#253で実装した」ではなく「弱点起点の復習」のように、機能や学習体験上の役割を中心に記述する。

#### 初見読者がIssue階層を知らなくても読めるようにする

本文は、Issue階層や過去の議論を知らない読者でも、弱点分析の主要な意味、機能関係、設計思想を理解できる自己完結性を持たせる。

必要な背景は本文中で要約し、詳細な判断経緯や受入条件を確認したい場合にIssueやPRを参照できる構成にする。

### 目次設計で使った判断基準

目次案比較では、次の観点を判断基準にした。

- 初見読者が理解しやすいか。
- Issue階層を知らなくても読めるか。
- 実装目的が中心になっているか。
- 学習フィードバックループが理解できるか。
- 詳細仕様の再掲になっていないか。
- 章間重複を避けられるか。
- 現行実装と将来展望を分離できるか。
- 今後の分析・可視化・学習支援の展望まで自然につながるか。
- 後続の章別詳細化へ分割しやすいか。

この判断基準に基づき、学習体験とプロダクト概念を中心にした章構成を採用する。

## 第1章 弱点分析の位置付け

本章では、弱点分析が必要になった背景と、クイズを「解く・保存する」体験から、学習履歴を使って次の学習行動を決める体験へ拡張した意味を説明する。

DEPクイズアプリの基本的な学習体験は、問題を解き、その結果を保存し、学習履歴を蓄積することから始まる。この流れにより、利用者は自分がどの問題に取り組み、どのような結果を残したかを後から確認できるようになった。学習履歴は、単発の解答結果を失わずに積み上げるための土台であり、継続的な学習を支える重要な情報である。

しかし、履歴が蓄積されているだけでは、次に何を学ぶべきかを判断するには十分ではない。正解数や過去の解答結果を見られても、どの範囲でつまずきが多いのか、どの誤答傾向を優先して復習すべきか、今の学習で注目すべき対象は何かを利用者が自力で読み解く必要が残る。つまり、「解く」「保存する」だけでは、学習履歴が次の学習行動へ自然につながり切らないという課題があった。

弱点分析は、この課題を解決するために、蓄積された学習履歴を振り返りの材料から学習判断の材料へ変える役割を持つ。学習履歴を分析することで、利用者は単に過去の結果を見るのではなく、自分がどの観点を重点的に見直すべきかを把握しやすくなる。ここで重視したのは、分析結果そのものを目的にすることではなく、分析を通じて次の学習行動を決めやすくすることである。

この考え方により、学習体験は、問題を解いて結果を保存するだけの流れから、蓄積された履歴を分析し、弱点を見つけ、次の学習行動を決める流れへ拡張される。図1は、この変化を「従来の学習体験」と「弱点分析を導入した学習体験」の対比として示している。

![学習履歴を次の学習判断に変える弱点分析の位置付け](images/phase-d-weakness-learning-section1.png)

弱点分析によって、学習履歴は「過去の結果の記録」から「次に取り組むべき内容を考えるための入口」へ変わる。図のように、利用者は、蓄積された結果を手がかりに弱点を見つけ、復習の優先度を考え、次の学習へ進むことができる。

一方で、弱点分析の結果を見るだけでは、学習行動は完結しない。分析で弱点が見えても、どの問題が対象なのかを確認できなければ復習に移りづらく、復習した結果が学習履歴へ戻らなければ改善状況を再び判断できない。また、学習を一区切りにして新しい周期へ移りたい場面では、履歴をどのように扱うかも学習体験の一部になる。

そのため、弱点分析は単独の閲覧機能ではなく、対象確認、弱点起点の復習、進捗更新、必要に応じた学習履歴リセットと組み合わせて意味を持つ。第2章では、これらの機能がどのように接続され、学習履歴を次の行動へ変えるフィードバックループを形成するかを説明する。

## 第2章 弱点分析を起点とする学習フィードバックループ

本章では、分析、対象確認、復習、進捗更新、再分析、必要に応じた学習履歴リセットが、一つの学習循環としてどう接続されるかを説明する。

弱点分析は、学習履歴から弱点を見つける入口である。ただし、分析結果を表示するだけでは、利用者が実際に復習し、進捗を更新し、改善後の状態を確認するところまでは到達しない。弱点分析を学習行動へつなげるには、分析結果の確認から復習、履歴更新、再分析までを一連の流れとして扱う必要がある。

この流れは、図2のように、学習履歴を起点に弱点分析、対象確認、弱点起点の復習、進捗更新、再分析へ戻る学習フィードバックループとして整理できる。図は、通常の復習サイクルと、必要に応じて学習履歴リセットから新しい学習周期へ移る分岐を一体で示している。

![弱点分析から対象確認、復習、進捗更新、再分析へつながる学習フィードバックループ](images/phase-d-weakness-learning-section2.png)

まず、既存の学習履歴が弱点分析の入力になる。分析は、蓄積された結果をもとに、利用者が次に注目すべき弱点を判断するための情報を提示する。ここでの分析結果は、復習へ進むための出発点であり、最終的な到達点ではない。

次に、対象確認が分析と復習の間をつなぐ。分析結果だけでは、利用者が具体的にどの問題を復習するのかを十分に把握できない場合がある。対象問題一覧を通じて、弱点として扱われた問題群を確認できるようにすることで、利用者は復習対象に納得したうえで次の行動へ進める。対象確認は、分析結果を抽象的な傾向から具体的な復習対象へ橋渡しする役割を持つ。

その後、弱点起点の復習によって、利用者は確認した対象に沿って再度問題に取り組む。復習の結果は、独立した一時的な記録として閉じるのではなく、既存の学習履歴へ戻る。これにより、復習で得られた正誤や進捗が次回以降の判断材料に含まれ、最新の履歴をもとに再び弱点分析できる。

この循環により、弱点分析は一度だけ見るレポートではなく、学習履歴を更新しながら繰り返し利用する判断の起点になる。利用者は、分析で弱点を見つけ、対象を確認し、復習し、その結果を履歴へ戻し、更新された履歴から再分析することで、学習状態の変化を次の行動へ反映できる。

学習履歴リセットは、この循環を毎回通過する通常工程ではない。リセットは、学習を一区切りにしたい場合や、蓄積された履歴を前提にした判断から新しい学習周期へ切り替えたい場合に使う分岐である。したがって、弱点分析から常にリセットへ進むのではなく、通常は対象確認と復習を通じて履歴を更新し、必要な場面でだけ新しい周期への切替手段として利用する。

このように、弱点分析、対象問題一覧、弱点起点の復習、学習履歴リセットは、ばらばらに追加された機能ではない。それぞれが、学習履歴を起点にして、弱点を見つけ、復習し、更新された状態を再び判断するためのフィードバックループを構成している。

次章では、このループの起点である弱点分析が、蓄積された学習履歴をどのように学習判断へ変換するのかを説明する。

## 第3章 学習履歴を学習判断へ変える ― 弱点分析の仕組み

学習履歴だけでは、次に何を見るべきか判断しづらい。弱点分析は、蓄積された履歴を次の学習判断へ変えるための入口である。

![学習履歴を学習判断へ変換する弱点分析の仕組み](images/phase-d-weakness-learning-section3.png)

図3では、学習履歴を複数の観点から整理し、判断材料へ変換する流れを示している。読み手がここで押さえるべき点は、低い正答率を機械的に同じ弱点として扱わないことである。未学習、データ不足、実際に誤答が残る弱点は学習上の意味が異なるため、区別して判断できることが重要になる。

また、弱点分析は固定された成績表ではない。復習や通常学習によって履歴が更新されれば、次に見るべき内容も変わるため、常に最新履歴から現在の判断を導く。

ただし、判断材料が得られても、それだけでは復習行動に移れない。次章では、分析結果を具体的な対象確認と復習へつなぐ役割を説明する。

## 第4章 分析を学習行動へつなぐ ― 対象問題一覧と弱点起点の復習

分析結果を見るだけでは、学習行動にはならない。弱点として示された内容を、利用者が確認できる具体的な復習対象へつなぐ必要がある。

![分析結果を対象問題一覧と弱点起点の復習へつなぐ流れ](images/phase-d-weakness-learning-section4.png)

図4では、分析結果から対象問題一覧、弱点起点の復習、進捗反映へ進む流れを示している。この流れで特に押さえるべき点は、対象問題一覧を分析と復習の橋渡しとして置くことである。利用者は、弱点条件に対応する問題群を確認することで、分析結果を「実際に取り組む問題」として受け止められる。

復習結果は専用の一時履歴に閉じず、既存の学習履歴へ戻す。これにより、弱点起点で解いた結果が次回以降の分析材料に含まれ、復習後の状態を再分析できる。

一方で、履歴を更新し続けるだけでなく、試験前後や学習方針の変更に合わせて一区切りにしたい場面もある。次章では、新しい学習周期へ移るための履歴リセットを説明する。

## 第5章 学習サイクルを切り替える ― 学習履歴リセット

学習履歴リセットは、すべてを消す操作ではなく、これまでの履歴を一区切りにして新しい学習周期へ移るための手段である。

![学習履歴を一区切りにし学習資産を残して新しい周期へ移るリセットの考え方](images/phase-d-weakness-learning-section5.png)

図5では、評価履歴を区切りながら、学習資産を残して次の周期へ移る考え方を示している。ここで重要なのは、評価履歴と学習資産を同じものとして扱わないことである。学習結果や誤答理由は現在の状態を判断する材料であり、新しい周期へ切り替えるときには一区切りにする。

一方で、利用者が育てたメモ、ブックマーク、設定のような学習資産は、新しい周期でも足場として再利用する。履歴は区切り、資産は再利用する。この考え方は、弱点分析、復習、リセットを安全に循環させるための設計思想につながる。次章では、その主要な設計思想を整理する。

## 第6章 弱点分析を支える主要な設計思想

本章では、弱点分析、対象問題一覧、弱点起点の復習、学習履歴リセットに共通する設計思想を整理する。第3章から第5章では各機能が学習循環の中で果たす役割を説明した。本章で扱うのは、それらを個別機能としてではなく、同じ状態管理と学習体験の上に安全に接続するために、なぜ現在の方針を採ったのかである。

弱点分析の機能群は、状態を重複して持たず、最新状態から必要な情報を導出し、既存の学習基盤を再利用し、閲覧と状態変更を分離することで成立している。この方針により、分析、対象確認、復習、再分析、必要に応じた履歴リセットが、互いに矛盾しない一つの学習循環として扱える。

| 設計思想                   | 解決したかった問題                                                                         | 採用した方針                                                                                                 | 得られる効果                                                                 |
| -------------------------- | ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------- |
| 既存状態を正本にする       | 弱点分析専用の状態を増やすと、学習履歴・設定・進行中セッションとの整合性を保ちにくくなる。 | 学習結果、設定、進行中の学習は、既存状態を各責務の正本として利用する。                                       | 状態の二重管理を避け、分析・復習・再開の前提をそろえやすくなる。             |
| 最新履歴から再導出する     | 保存済みの分析スナップショットを持つと、通常学習や復習後に分析結果が古くなる。             | 分析結果そのものは固定保存せず、必要なときに最新の履歴から導く。                                             | 復習や通常学習で変化した状態を、次回の弱点分析へ反映できる。                 |
| 既存クイズ体験を再利用する | 弱点復習専用のクイズエンジンを作ると、解答、保存、再開、完了の体験が通常学習と分断される。 | 復習対象の選び方だけを弱点条件に合わせ、問題を解く体験は既存基盤を使う。                                     | 利用者は一貫した操作感で復習でき、実装上も既存の保存・再開の責務を活かせる。 |
| 閲覧と状態変更を分離する   | 分析表示や対象確認だけで状態が変わると、利用者が意図しない更新やリセットを起こし得る。     | 分析表示、対象確認、条件再選択は状態を変更せず、復習開始、回答、履歴リセットなど明示的な操作だけで更新する。 | どの操作が安全な閲覧で、どの操作が状態変更かを予測しやすくなる。             |

第一に、既存状態を正本にすることで、弱点分析は独立した別系統のデータ管理にならない。学習履歴は現在の理解度を判断する材料であり、設定は問題の出題や学習条件を決める前提であり、進行中セッションは中断・再開を含む学習体験の現在地である。これらを弱点分析用に複製すると、どちらが正しい状態なのかを常に調整する必要が生じる。現在の設計では、弱点分析は既存状態を読み取り、復習開始時には既存の進行中セッションの仕組みに接続することで、正本を分散させない。

第二に、分析結果を保存せず最新履歴から再導出することで、弱点分析は過去の固定レポートではなく、現在の学習状態を映す入口になる。復習や通常学習によって正誤や誤答理由が変われば、次に見るべき弱点も変わる。分析結果を独立したスナップショットとして保存すると、履歴更新後に古い分析を見続ける危険があるため、必要な情報は最新履歴から導く方針にしている。

第三に、既存のクイズ体験を再利用することで、弱点起点の復習を新しい学習モードとして孤立させない。弱点復習で変えるべきなのは、どの問題を対象にするかであり、解答する、結果を保存する、中断して再開する、完了後に結果を見るという基本体験そのものではない。対象選択だけを弱点条件に合わせ、実際の学習体験は既存基盤へ戻すことで、通常学習と弱点復習が同じ履歴更新の循環に入る。

第四に、閲覧と状態変更を分離することで、分析と確認を安心して行える。利用者は、弱点分析を開き、対象問題を確認し、別条件を選び直す段階では、まだ学習履歴や進行中セッションを変えたくない場合がある。状態が変わる境界を、復習開始、回答保存、履歴リセットのような明示的な操作に限定することで、分析画面は判断のための安全な閲覧場所になり、状態変更は利用者が意図した操作として扱える。

これらの思想は、整合性、鮮度、再利用性、安全性を分担して支えている。既存状態を正本にすることで整合性を保ち、最新履歴から再導出することで分析の鮮度を保ち、既存クイズ体験を再利用することで学習体験を分断せず、閲覧と状態変更を分離することで利用者の予測可能性を高める。第1章から第5章で説明した機能群は、この横断的な判断の上で一つの弱点学習体験として接続されている。

次章では、この設計思想によって現在どこまで実現できているかを整理し、まだ現在の機能として扱わない領域と、今後発展し得る方向を分けて説明する。

## 第7章 現在地と今後の発展方向

本章では、現在実現できている分析・復習支援と、今後発展し得る分析・可視化・学習支援を明確に分けて整理する。弱点分析は、学習履歴を次の行動へつなげる基盤を作った段階にある。一方で、長期的な変化の可視化や自動推薦までを現在の機能として含めているわけではない。

### 現在実現できていること

現在の弱点分析では、学習履歴を複数の観点から分析し、利用者が次に注目すべき対象を判断しやすくしている。全体の状況だけでなく、Sectionや誤答理由タグの観点から弱点を見られるため、単なる正答率確認ではなく、どの範囲や傾向を優先して見直すかを考えやすい。

分析結果は、対象問題一覧へ接続されている。利用者は、弱点として示された条件に該当する問題を確認し、復習対象を具体的な問題群として把握できる。これにより、分析結果を抽象的な傾向のまま終わらせず、実際に取り組む対象へ変換できる。

弱点条件を起点にした復習では、対象の選び方を弱点分析に合わせつつ、問題を解く体験は既存のクイズ体験に接続している。復習で得られた結果は既存の学習履歴へ戻るため、復習後に再び弱点分析を見ると、最新状態にもとづく判断へ進める。

また、学習履歴リセットにより、評価履歴を一区切りにして新しい学習周期へ移る手段も用意している。これは、これまでの学習資産をすべて捨てるための操作ではなく、評価に使う履歴を切り替えながら、メモ、ブックマーク、設定のような学習の足場を残すための境界である。

### 現在の境界・未実装領域

現在の弱点分析は、最新の学習履歴から現在の弱点を導くことを中心にしている。そのため、週ごと、月ごと、学習周期ごとの変化を時系列で比較し、弱点がどのように改善または悪化したかを示す機能は扱っていない。

また、Sectionと誤答理由タグなど、複数観点を組み合わせた高度なクロス分析は現在の範囲外である。現在は、利用者が複数の観点を見比べて判断できる状態を作ることを優先しており、組み合わせ条件による詳細な分析モデルまでは定義していない。

復習前後の学習効果を、独立した指標として測定する機能も現在の範囲には含めていない。復習結果は学習履歴へ戻り、次回の分析材料にはなるが、「復習によってどれだけ改善したか」を専用指標として算出する設計にはしていない。

AIによる自動推薦や学習計画生成も、現在実装済みの機能ではない。現時点では、弱点分析が判断材料を提示し、利用者が対象確認や復習開始を選ぶ構造である。自動的に優先順位や学習順序を決定することは、十分な履歴、評価方法、説明可能性を検討した後の候補として分けて扱う。

### 今後の発展方向

今後の発展方向としては、まず時系列での弱点・改善傾向の把握が考えられる。現在の分析が「今どこに弱点があるか」を示すのに対し、時系列分析は「以前と比べてどう変わったか」を示す。これにより、学習の継続による変化を利用者が振り返りやすくなる可能性がある。

次に、複数観点を組み合わせたクロス分析が考えられる。Sectionと誤答理由タグ、正答率と試行回数、復習頻度と改善状況などを組み合わせることで、単一観点では見えにくい学習課題を扱える可能性がある。ただし、どの組み合わせが有用か、利用者に過剰な複雑さを与えないかを検討する必要がある。

復習前後の学習効果測定も発展候補である。復習結果を履歴へ戻す現在の仕組みを土台に、復習前後の変化を比較できれば、利用者は取り組みの効果をより明確に把握できる。ただし、改善をどの期間・どの母数・どの指標で測るかは、別途設計が必要である。

さらに、十分な履歴と評価方法が整った場合には、優先順位や学習順序の支援、AIによる推薦や学習計画支援へ発展する余地がある。これらは現在の確定仕様や実装予定ではなく、弱点分析が蓄積する判断材料を将来どのように活かし得るかという候補である。

| 領域 | 現在地                                                                      | 今後の発展方向                                                       |
| ---- | --------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| 分析 | 最新履歴から、全体・Section・誤答理由タグなどの観点で現在の弱点を把握する。 | 時系列分析や複数観点のクロス分析により、変化や組み合わせ傾向を扱う。 |
| 復習 | 弱点条件に該当する対象を確認し、既存クイズ体験で復習する。                  | 優先順位や学習順序の支援により、取り組み方の判断を補助する。         |
| 評価 | 復習結果を既存の学習履歴へ反映し、最新状態から再分析する。                  | 復習前後の変化を比較し、学習効果を測定する。                         |
| 推薦 | 利用者が分析結果と対象確認にもとづいて次の行動を選ぶ。                      | 十分な履歴と評価方法を前提に、AI推薦や学習計画支援を検討する。       |

このように、現在の弱点分析は、最新履歴から弱点を見つけ、対象を確認し、復習し、その結果を再分析へ戻す循環を実現している。一方で、長期変化の分析、高度なクロス分析、効果測定、自動推薦は将来候補であり、現在の機能として断定しない。次章では、現在地や将来候補の根拠、実装、保証内容を確認するために、どの正本を参照すべきかを整理する。

## 第8章 参考情報

本章では、詳細な要件、判断経緯、実装契約、テスト保証を確認するために、どの成果物を参照すべきかを示す。本文書は、弱点分析の意味、目的、機能関係、設計思想、現在地を理解する入口である。具体的な要件、変更差分、現在の挙動、保証される動作は、目的に応じて各正本を確認する。

| 実際にどの変更単位で統合されたかを確認したい。 | PR                        | レビュー単位、変更範囲、確認結果を確認する。                                         | 各機能を実装・文書化した代表PR。                                                                          |

Issueは、背景、要件、判断経緯、受入条件を確認するための正本である。弱点分析がなぜ必要になったのか、対象問題一覧や弱点起点の復習がどの課題を補うために追加されたのか、学習履歴リセットがどの境界を扱うのかを確認したい場合は、関連Issueを参照する。本文書では、Issue本文を時系列に再掲するのではなく、機能の意味として要約している。

PRは、実際の変更単位を確認するための正本である。ある機能がどの差分としてレビューされ、どの確認結果とともに統合されたかを見たい場合は、PRを参照する。本文書はPR差分の一覧ではないため、実装ファイルごとの変更内容やレビュー履歴はPR側で確認する。

コードは、現在の挙動を確認するための正本である。分析がどの状態から導出されるか、弱点復習のセッションがどのように作られるか、保存や再開がどの既存経路に接続されるかは、最終的にはコードで確認する。本文書では設計思想を理解するために実装名を扱うことがあるが、APIやstorage契約の完全な仕様はコードを正とする。

Unit TestとE2Eは、保証される動作を確認するための正本である。特定の入力でどの分析結果になるか、表示や操作がどの境界を守るか、閲覧だけではstorageを変更しないか、復習結果が履歴へ戻るかといった回帰保護の内容は、テストで確認する。E2Eの責務や重複を避けた保証範囲を把握したい場合は、E2Eカタログを入口にする。

機能実装書である本文書は、これらの正本を読む前に、弱点分析の全体像をつかむための入口である。読者は、まず本文書で、弱点分析がなぜ必要で、どの機能がどのように接続され、どの設計思想によって支えられ、現在どこまで実現されているかを把握する。そのうえで、確認したい内容に応じてIssue、PR、コード、Unit Test、E2E、E2Eカタログへ進む。

本文書は、機能の意味と設計意図を理解する入口であり、具体的な変更や保証内容は、目的に応じてIssue・PR・コード・テストなどの正本を参照する。
