# 2. サービス体系と学習導線

## 各サービスの役割

### 学習者向けサービス

- 本プラットフォームのフロントサービスである
- 学習効果を高めること、その習熟度を計測するためのサービスを提供する

| サービス      | 主な役割                           | 利用者が得るもの                       | 主な成果物                       |
| ------------- | ---------------------------------- | -------------------------------------- | -------------------------------- |
| Audio Learn   | 概念・背景・全体像を理解する       | なぜ必要か、何のためかが分かる         | 短尺音声、要点、キーワード       |
| Quiz Practice | 知識・判断ポイントを確認する       | 何を選ぶべきか、どこが弱いかが分かる   | 設問、解説、誤答理由、復習導線   |
| Mission Lab   | 操作・結果確認・失敗対応を体験する | どう実行するか、何を確認するかが分かる | 模擬操作、分岐、ヒント、完了条件 |

### 習熟度分析者向けサービス

- 本プラットフォームのバックエンドサービスである
- 学習効果や習熟度を可視化することで意思決定に寄与することを目的とする

| サービス            | 主な役割                   | 利用者が得るもの                         | 主な成果物                 |
| ------------------- | -------------------------- | ---------------------------------------- | -------------------------- |
| Readiness Dashboard | 習熟度や進捗度を可視化する | 稼働判定、学習プランの意思決定材料を得る | ダッシュボード、学習データ |

### コンテンツ実装者向けサービス

- 本プラットフォームのドライバサービスである
- コンテンツを早期かつ簡素に立ち上げ可能にすることを目的とする

| サービス                    | 主な役割                                 | 利用者が得るもの                     | 主な成果物         |
| --------------------------- | ---------------------------------------- | ------------------------------------ | ------------------ |
| Learning Experience Factory | 学習者向けサービスを簡素に整備可能にする | 教育環境の迅速な立ち上げが可能になる | 学習者向けサービス |

## 基本学習導線

```text
Audio Learn
概念・背景を理解する
        ↓
Quiz Practice
判断ポイント・誤解しやすい点を確認する
        ↓
Mission Lab
操作し、結果を確認し、失敗も体験する
        ↓
振り返り
弱点に応じてAudio / Quizへ戻る
```

## 学習者の状態別導線

| 学習者の状態           | 推奨導線                        | 主な狙い                                     |
| ---------------------- | ------------------------------- | -------------------------------------------- |
| 初学者                 | Audio → Quiz → Mission          | 概念から実践まで段階的に理解する             |
| 試験対策中             | Quiz → Audio補強 → Quiz         | 知識の穴を効率よく補強する                   |
| 実務経験者             | Mission → Quiz → Audio補足      | 操作・判断の確認を中心に進める               |
| 新運用・変更対象者     | Audio → Mission → Quiz          | 新ルールを理解し、安全に実行できるようにする |
| 苦手テーマがある学習者 | Quiz / Mission → Audio → 再挑戦 | 弱点を起点に再学習する                       |

## サービス連携の原則

- Audio、Quiz、Missionは別サービスではなく、同一テーマを異なる角度から学ぶ一連の体験として扱う。
- 各サービスは共通のテーマID、ロール、業務、判断ポイントを参照する。
- QuizまたはMissionで弱点が検出された場合、関連Audioまたは再挑戦すべきMissionへ戻す。
- 学習の評価は受講率だけでなく、知識・判断・操作の複合で扱う。

# 3. 学習テーマ体系（Learning Taxonomy）

- 本章は、本Learning Hubが「何を学べるようにするか」を定義するコンテンツの全体地図である。
- Learning HubにおけるSectionは、試験名称ではなく、実務・業務上も通用する能力領域として定義する。

## 階層定義と設計原則

```text
Section
  └─ Chapter / Category
       └─ 学習テーマ一覧
            └─ Learning Portfolio
                 └─ Source Pack
                      └─ Audio / Quiz / Mission
```

| レベル             | 定義                               | 主な問い                               | 例                               |
| ------------------ | ---------------------------------- | -------------------------------------- | -------------------------------- |
| Section            | 実務上通用する大きな能力領域       | どの領域の能力を育てるか               | データ開発と取り込み             |
| Chapter / Category | まとまりのある能力・業務カテゴリ   | 何を一連の能力として身につけるか       | ファイル取り込みとメダリオン設計 |
| Learning Theme     | 一つの学習体験として成立するテーマ | 何を理解・判断・操作できるようにするか | CSVをBronzeからSilverへ届ける    |

### 設計原則

- Sectionは、試験固有の名称ではなく、導入後の業務・運用でも通用する能力領域として命名する。
- 既存のQuizやAudioの7セクションは廃棄せず、`reference_framework` として対応関係を持たせる。
- Themeは概念名ではなく、可能な限り「業務・判断・操作・到達状態」が見える表現にする。
- 一つのThemeは、原則としてAudio、Quiz、Missionの少なくとも一つに展開できる粒度とする。
- Themeを増やす前に、既存Section・Chapterとの重複や関係を確認する。

### テーマ体系の更新原則

- 新しいThemeは、必ずSectionとChapterに所属させる。
- 既存のThemeと重複する場合は、新設ではなく統合・分岐・拡張を検討する。
- 試験範囲の変更、製品機能の追加、業務運用の変更は、Section / Chapter / Themeへの影響を確認する。
- AudioやQuizだけで先行して存在するテーマも、順次Taxonomy上の位置付けを付与する。
- Themeの追加・統合・廃止は、Learning Portfolioの優先度見直しと合わせてレビューする。

# 4. Learning Portfolio

- 本章は、Learning Taxonomyに定義されたテーマ群のうち、誰に・どの業務に・何を優先して提供するかを決める投資判断の地図である。

## 対象ロール一覧

| ---------------- | ----------------------------------- | ---------------------------------------- |
| データエンジニア | 取り込み、変換、品質確認、運用      | データ処理を安全に実装・確認できる       |
| 分析者           | データ探索、SQL利用、公開データ活用 | ガバナンスを守りながらデータを活用できる |
| 基盤管理者       | 権限、ガバナンス、監査、運用管理    | 安全な利用環境を設計・維持できる         |

| ---------------- | ----------------------------------------- | ---------- | -------- | ---------- | ----------------- |
| データエンジニア | CSVを取り込み、BronzeからSilverへ変換する | 高         | 高       | 最優先     | Mission 01候補    |
| データエンジニア | スキーマ不一致を検知し、適切に対応する    | 高         | 高       | 高         | 運用系Mission候補 |

## テーマ優先順位の判定基準

各テーマは、以下の観点で優先順位を判断する。

| 観点       | 判断基準                                          |
| ---------- | ------------------------------------------------- |
| 業務重要度 | 本番業務における頻度・影響度が高いか              |
| 失敗リスク | 誤操作や判断ミスが障害・品質事故につながるか      |
| 学習効果   | Audio / Quiz / Missionの3サービスに展開しやすいか |
| 既存成果物 | 設計書、手順書、Issue、FAQ、実装例が活用できるか  |
| 横展開性   | 他のテーマや将来の導入PJにも再利用しやすいか      |
| 更新頻度   | 変更に追従可能な粒度に収まっているか              |

# 5. 本設計書と後続成果物の関係

```text
Learning Hub全体設計書
 ├─ Learning Taxonomy
```

## 本設計書の更新契機

- 新しい対象ロールが追加された
- 重要業務や業務フローが変更された
- 新しいサービスが追加された
- 学習データから優先テーマの見直しが必要になった
- システムの導入・運用方針が大きく変わった

# 6. 承認・レビュー観点

| 観点         | 確認内容                                             |
| ------------ | ---------------------------------------------------- |
| 役割整合性   | 対象ロールと実際の責務が一致しているか               |
| 業務重要性   | Learning Portfolioが本番で重要な業務を捉えているか   |
| サービス分担 | Audio / Quiz / Missionの役割が重複・欠落していないか |
| 優先順位     | MVPテーマの選定理由が明確か                          |
| 実現可能性   | 既存成果物を活用して最初の縦型テーマを作れるか       |
| 将来拡張性   | 新テーマ・新ロール・新サービスを追加できる構造か     |
